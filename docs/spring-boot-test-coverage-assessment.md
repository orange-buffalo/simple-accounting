# Spring Boot Test Coverage Assessment

This inventory includes Spring Boot tests that extend `SaIntegrationTestBase`, excluding full stack tests that extend
`SaFullStackTestBase` and GraphQL API tests under `business/api` or `infra/graphql`. The two document upload/download
REST endpoint tests under `business/api/documents` are included because they are not GraphQL API tests.

| Test | Extends base test class | Coverage assessment |
| --- | --- | --- |
| `business/api/documents/DocumentsContentApiTest` | Yes, `SaIntegrationTestBase` | Partly covered by higher-level document download flows such as `DocumentsOverviewFullStackTest` and `DocumentDownloadByWorkspaceTokenFullStackTest`. The anonymous token endpoint and 404 mapping are REST boundary behavior; they could be covered through a full stack/browser download scenario, but that would be more brittle than this endpoint-level test. |
| `business/api/documents/DocumentsUploadApiTest` | Yes, `SaIntegrationTestBase` | Partly covered by full stack upload scenarios in `SaDocumentsUploadFullStackTest`, `SaDocumentsUploadLocalFileSystemFullStackTest`, `SaDocumentsUploadGoogleDriveFullStackTest`, and entity create/edit tests with attachments. Token-not-found, anonymous upload access, binary content type, and workspace association are lower-level REST upload contract checks; some could be covered higher, but with more setup and weaker failure localization. |
| `business/common/pesistence/TimestampsTest` | Yes, `SaIntegrationTestBase` | Not meaningfully covered by higher-level tests. It verifies database timestamp-with-time-zone persistence mapping; covering it through API or UI flows would be indirect and would not clearly isolate mapping regressions. |
| `business/documents/DocumentsServiceTest` | Yes, `SaIntegrationTestBase` | Partly covered by document storage status/download API and full stack flows. The service-level delegation checks are narrow; higher-level coverage is possible for user-visible behavior, but not for verifying the service selects the expected storage/provider path without extra test doubles. |
| `business/documents/storage/gdrive/GoogleDriveDocumentsStorageServiceTest` | Yes, `SaIntegrationTestBase` | Mostly covered by `UserProfileGoogleDriveDocumentStorageFullStackTest` plus Google Drive upload/download component tests. Remaining protocol details, push-notification behavior, and authorization failure handling depend on Google Drive/OAuth mocks and are harder to cover at full stack level without significant setup and flakiness. |
| `business/expenses/ExpenseServiceTest` | Yes, `SaIntegrationTestBase` | Representative scenarios are covered by `CreateExpenseMutationTest`, `EditExpenseMutationTest`, `CreateExpenseFullStackTest`, and `EditExpenseFullStackTest`. The full calculation matrix for default/foreign currency, business percentage, tax, and conversion status is not fully covered higher; adding all cases there would be slow and noisy, so service-level coverage is appropriate. |
| `business/generaltaxes/GeneralTaxesReportingServiceTest` | Yes, `SaIntegrationTestBase` | Covered at a higher level by `ReportingFullStackTest`, which exercises general tax report combinations, finalized-only behavior, and pending items. This service test appears largely redundant unless it is kept as a fast diagnostic for report calculation regressions. |
| `business/incomes/IncomesServiceTest` | Yes, `SaIntegrationTestBase` | Representative scenarios are covered by `CreateIncomeMutationTest`, `EditIncomeMutationTest`, `CreateIncomeFullStackTest`, `EditIncomeFullStackTest`, and `IncomesOverviewFullStackTest`. The detailed conversion/tax matrix and linked-invoice side effects are not exhaustively covered higher; full higher-level duplication would add considerable cost. |
| `business/integration/downloads/DownloadsServiceTest` | Yes, `SaIntegrationTestBase` | Mostly covered by `CreateDocumentDownloadUrlMutationTest`, `DocumentsContentApiTest`, and full stack document download tests. Unknown-token and provider delegation can be covered at API level, but this service test gives cheaper and clearer coverage of token resolution internals. |
| `business/invoices/InvoicesServiceTest` | Yes, `SaIntegrationTestBase` | Partly covered by `CreateInvoiceMutationTest`, `EditInvoiceMutationTest`, `CancelInvoiceMutationTest`, `InvoicesQueryTest`, and invoice full stack tests. Overdue transition logic is business-rule heavy and not fully exercised higher; it could be covered via API query/update scenarios, but a focused service test is lower cost and easier to diagnose. |
| `business/security/authentication/BruteForceDefenseTest` | Yes, `SaIntegrationTestBase` | Basic bad-credentials behavior is covered by `CreateAccessTokenByCredentialsMutationTest`. Progressive lock timing, lock expiration boundaries, cap behavior, and parallel request throttling are not fully covered higher; they are technically possible through GraphQL login tests, but would be slower and more timing-sensitive. |
| `business/security/jwt/RefreshTokensServiceTest` | Yes, `SaIntegrationTestBase` | Covered in user-visible form by `RefreshAccessTokenMutationTest` and `InvalidateRefreshTokenMutationTest`. Token generation, persistence lookup, expiration, and prolongation internals are not all covered higher; keeping focused service coverage avoids forcing cookie/API tests to validate persistence details. |
| `business/ui/WebUiConfigTest` | Yes, `SaIntegrationTestBase` | Full stack tests implicitly verify that the app shell loads. Favicon/static resource serving, SPA route fallback, anonymous access, and cache headers are HTTP configuration details not directly asserted higher; they can be covered with endpoint tests more cleanly than browser tests. |
| `business/users/migration/AdminUserRecoveryServiceTest` | Yes, `SaIntegrationTestBase` | Not covered by higher-level tests. It verifies startup/recovery data migration behavior for initial admin creation; exercising this through UI/API would require specialized application bootstrap states and would be less direct. |
| `business/workspaces/WorkspacesServiceTest` | Yes, `SaIntegrationTestBase` | Partly covered by workspace GraphQL tests and full stack workspace-token flows such as `WorkspaceTokenAccessWalkThroughFullStackTest` and `WorkspacesOverviewFullStackTest`. The complete access-mode matrix across owner, regular users, transient workspace-token users, expired tokens, revoked tokens, and other workspaces is not fully covered higher; duplicating the matrix at API/UI level would be expensive. |
| `infra/oauth2/DbReactiveOAuth2AuthorizedClientServiceTest` | Yes, `SaIntegrationTestBase` | Indirectly covered by Google Drive profile and OAuth full stack flows. The persistence details for authorized clients, including refresh-token preservation on updates, are not easily asserted at a higher level without coupling tests to infrastructure state. |
| `infra/oauth2/OAuth2ClientAuthorizationProviderTest` | Yes, `SaIntegrationTestBase` | Partly covered by `CompleteOAuth2FlowMutationTest` and Google Drive profile authorization flows. Authorization URL construction, token endpoint parameters, event emission, and failure modes are infrastructure contracts; higher-level coverage is possible for happy/error flows but would not validate protocol details cleanly. |
| `infra/oauth2/OAuth2WebClientBuilderProviderTest` | Yes, `SaIntegrationTestBase` | Indirectly covered by Google Drive storage/profile flows. Access-token reuse, missing authorization cleanup, and refresh-token renewal are infrastructure behaviors that are difficult to verify through UI/API tests without broad mocks and timing-sensitive setup. |
